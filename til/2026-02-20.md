## Today I Learned

### git, github, pr 기본 흐름

#### git이랑 github 차이
- git은 내 로컬에서 버전 관리하는 도구, github는 그 git 저장소를 원격으로 공유하고 협업하는 플랫폼.
- 혼자 개발이면 git만으로도 되는데, 팀 개발은 github 이슈, PR, 리뷰, 권한 같은 협업 기능이 핵심.

#### PR(Pull Request)이 뭐냐
- 내가 만든 변경사항을 메인 브랜치에 합치기 전에, 코드 리뷰와 CI 체크를 받는 협업 단위.
- 그냥 “합쳐도 되나요?” 요청서라고 보면 편함.

PR 기본 흐름(내가 자주 하는 방식)
- main에서 feature 브랜치 땀
- 커밋 쌓음
- push 하고 PR 올림
- 리뷰 반영(추가 커밋 또는 rebase)
- 머지(merge 옵션 선택)
- 브랜치 정리(삭제)

### git 명령어(자주 쓰는 것 위주)

#### 상태 확인과 커밋
- git status: 지금 뭐가 바뀌었는지 확인
- git diff: 변경 내용 확인
- git add / git commit: 스테이징 후 커밋 만들기
- git log --oneline --graph: 커밋 히스토리 빠르게 보기

#### 브랜치 작업
- git branch / git switch -c <branch>: 브랜치 만들고 이동
- git switch <branch>: 브랜치 이동
- git merge <branch>: 브랜치 합치기(머지 커밋이 생길 수도 있음)

#### 원격 저장소
- git remote -v: 원격 주소 확인
- git fetch: 원격 변경사항만 가져오기(내 브랜치에 적용은 안 함)
- git pull: fetch + merge(or rebase)까지 같이
- git push: 내 커밋 원격에 올리기

### git rebase 정확한 개념(중요)

#### rebase 한 줄 정의
- 내 브랜치의 “기준 베이스 커밋”을 다른 커밋으로 갈아끼우고, 내 커밋들을 그 위에 다시 얹는 작업.

#### merge랑 뭐가 다르냐(감각)
- merge는 두 갈래 히스토리를 합쳐서 “합친 기록”이 남는다.
- rebase는 내 커밋들이 “마치 처음부터 최신 브랜치에서 작업한 것처럼” 일자로 정리된다.

#### 왜 쓰냐
- PR 올릴 때 커밋 히스토리를 깔끔하게 만들고 싶을 때(리뷰하기 쉬움).
- 최신 main 변경사항을 내 브랜치에 반영하면서 머지 커밋을 남기고 싶지 않을 때.

#### rebase 할 때 가장 중요한 주의점
- 이미 다른 사람이 쓰는 공용 브랜치에서 rebase 하면 커밋 해시가 바뀌어서 사고 난다.
- 내 개인 feature 브랜치에서만 하는 게 안전하다.
- rebase 후에는 보통 강제 push가 필요할 수 있다(강제 push는 조심해서).

#### 실무에서 많이 쓰는 형태
- git fetch origin
- git rebase origin/main
- 충돌 나면 해결하고 git add
- git rebase --continue
- 필요하면 git push --force-with-lease

### git flow vs github flow 차이

#### git flow
- 브랜치 종류가 많다: main(master), develop, feature, release, hotfix
- 릴리즈 관리가 분명하고, 배포 주기가 길거나 버전 관리가 강한 조직에서 쓰기 좋다.
- 대신 브랜치 전략이 무겁고, 팀이 익숙하지 않으면 운영 비용이 든다.

#### github flow
- main은 항상 배포 가능한 상태를 유지
- feature 브랜치 하나 파서 짧게 작업하고 PR로 리뷰 받고 바로 main에 합침
- CI/CD가 잘 되어 있고, 자주 배포하는 팀에 잘 맞는다.
- 규칙이 단순해서 온보딩이 쉽다.

### GitHub PR Merge 옵션 3가지

#### 1) Create a merge commit
- feature 브랜치와 main을 머지 커밋으로 합친다.
- 장점: 브랜치 작업 단위가 히스토리에 그대로 남아서 추적이 쉽다.
- 단점: 히스토리가 분기/머지로 복잡해질 수 있다.

#### 2) Squash and merge
- feature 브랜치의 여러 커밋을 “한 커밋”으로 뭉쳐서 main에 합친다.
- 장점: main 히스토리가 깔끔하고, 기능 단위로 보기 좋다.
- 단점: 작업 과정 커밋들이 사라져서 디테일 추적이 약해진다(대신 PR 기록으로 보완).

#### 3) Rebase and merge
- feature 커밋들을 main 위에 순서대로 다시 얹어서 일자 히스토리로 합친다.
- 장점: 히스토리 깔끔 + 커밋도 유지됨
- 단점: 커밋이 많고 충돌이 있으면 관리가 귀찮아질 수 있다.

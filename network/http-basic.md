# HTTP Basic

## 개요
HTTP는 웹에서 데이터를 주고받기 위한 통신 규약입니다. 브라우저(클라이언트)가 서버에 요청(Request)을 보내고, 서버가 응답(Response)을 보내는 형태로 동작합니다.

## 정의
- HTTP(HyperText Transfer Protocol): 데이터를 주고받는 양식을 정해둔 통신 규약
- 통신 규약: 컴퓨터끼리 데이터를 주고받을 때 지켜야 하는 약속
- Request: 클라이언트가 서버에 “이거 줘, 이거 해줘”라고 요청하는 메시지
- Response: 서버가 요청 처리 결과를 돌려주는 메시지

## 핵심 메커니즘(핵심 개념)
1) 요청과 응답 구조  
HTTP는 항상 Request와 Response 쌍으로 움직입니다.

2) 상태 코드(Status Code)  
응답에는 상태 코드가 있고, 요청이 성공했는지/실패했는지 등을 숫자로 표현합니다.

3) 헤더와 바디(페이로드)  
- Header는 부가 정보(메타 데이터)를 담습니다.
- Payload(Body)는 실제 데이터를 담습니다(HTML, JSON 등).

4) 메서드(Method)  
요청이 “무슨 행동”을 원하는지 표현합니다.

- GET: 조회(데이터 가져오기)
- POST: 생성/처리 요청(새 데이터 만들기, 서버에서 작업 수행)
- PUT: 전체 수정(리소스 전체를 새 값으로 교체하는 느낌)
- PATCH: 부분 수정(리소스의 일부만 변경)
- DELETE: 삭제
- HEAD: GET과 같지만 Body 없이 헤더만 받기(존재 여부/메타 정보 확인)
- OPTIONS: 이 URL에서 허용하는 메서드가 뭔지 확인(CORS에서 자주 보임)
- TRACE: 요청이 중간에 어떻게 전달되는지 추적(실무에선 거의 안 씀)
- CONNECT: 프록시를 통해 터널을 만들 때 사용(HTTPS 터널링 등)

## 동작 방식
- 브라우저가 URL로 서버에 요청을 보냅니다.
- 서버는 요청을 확인하고, 결과를 응답으로 반환합니다.
- 브라우저는 응답 데이터를 기반으로 화면을 그립니다.
- 이 과정은 개발자 도구(Network 탭)에서 직접 확인할 수 있습니다.

## 전체 흐름
1) 클라이언트가 HTTP Request를 만든다
2) 서버가 Request를 받고 처리한다
3) 서버가 HTTP Response를 만든다
4) Response에는 상태 코드, 헤더, (필요 시) 페이로드가 담긴다
5) 클라이언트가 Response를 받고 화면 렌더링 또는 데이터 처리한다

## 실무에서 고려할 포인트
- 문제 생겼을 때는 브라우저 개발자 도구(Network 탭)에서 요청/응답을 확인하면 원인 파악이 빨라집니다.
- 상태 코드만 봐도 실패 원인이 대략 분류됩니다(클라이언트 문제인지, 서버 문제인지).

## 용어 정리
| 용어 | 뜻 |
| --- | --- |
| HTTP | 웹에서 가장 널리 쓰는 통신 규약 |
| Request | 클라이언트가 서버로 보내는 요청 메시지 |
| Response | 서버가 클라이언트로 보내는 응답 메시지 |
| Method | 요청 목적(GET, POST 등) |
| Status Code | 요청 처리 결과를 나타내는 숫자(200, 404, 500 등) |
| Header | 부가 정보(메타 데이터) |
| Payload(Body) | 실제 데이터(HTML, JSON 등) |
| DevTools | 브라우저 개발자 도구(F12) |
| Network 탭 | HTTP 요청/응답 내역을 확인하는 탭 |

## 공부하면서 떠올릴 질문
- GET 요청에는 왜 보통 Body를 안 보낼까? (HTTP 약속 관점에서)
- 4xx와 5xx는 “누가 잘못한 건지” 관점에서 어떻게 구분하면 좋을까?
- Headers에는 어떤 정보들이 들어가고, 왜 필요한 걸까?
- Response의 Payload는 항상 있는 걸까? 없다면 어떤 상황일까?

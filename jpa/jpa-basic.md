# jpa-what-is-it-basic.md

## 개요
DB를 직접 다루는 방식(JDBC, SQL 직접 작성)은 반복 작업이 많고, 변경에 너무 취약합니다.
JPA는 이런 문제를 줄이기 위해 “객체(자바 클래스)” 중심으로 DB를 다루게 해주는 기술입니다.

## 정의
- ORM(Object Relational Mapping)
  - 객체(자바 클래스)와 관계형 DB(테이블)의 관계를 매핑해주는 기술입니다.
  - 객체를 저장/조회하면, 내부에서 SQL 실행과 결과 매핑 같은 작업을 대신 처리해주는 개념입니다.
- JPA(Java Persistence API)
  - 자바 ORM 기술에 대한 “표준 명세”입니다.
  - 즉, JPA는 제품이 아니라 “이렇게 만들어야 한다”는 규칙(스펙)에 가깝습니다.
- Hibernate(하이버네이트)
  - JPA 표준을 실제로 구현한 구현체(프레임워크) 중 사실상 표준(de facto)입니다.
  - 스프링 부트는 기본적으로 하이버네이트 구현체를 사용합니다.

## 핵심 개념
- JDBC로 직접 개발할 때의 문제
  - 테이블 만들기
  - SQL 직접 작성(INSERT, SELECT 등)
  - JDBC로 실행
  - 결과를 다시 객체로 수동 매핑(ResultSet to DTO)
  - 이 작업들이 계속 반복됩니다.
- SQL 의존적이라 변경에 취약
  - 예를 들어 Memo에 password 필드 하나 추가해도
    - INSERT SQL 수정
    - SELECT 결과 매핑 코드 수정
    - DTO 수정
  - “비즈니스 개발”보다 “SQL 유지보수”에 시간을 더 쓰게 되는 상황이 생깁니다.
- JPA가 해결하려는 방향
  - 객체 중심으로 다루면, 반복되는 SQL 작성과 결과 매핑 부담이 줄어듭니다.
  - DB 작업을 “간접적으로” 객체를 통해 처리합니다.

## 동작 방식
- JPA는 애플리케이션과 JDBC 사이에서 동작합니다.
- 개발자는 객체(Entity)를 다루지만,
  - 내부에서는 JPA 구현체(보통 Hibernate)가 JDBC를 통해 SQL을 실행합니다.
- 즉, 흐름은 이런 느낌입니다.
  - 애플리케이션(객체 조작)
    -> JPA(ORM, 표준)
      -> Hibernate(구현체)
        -> JDBC
          -> DB

## 전체 흐름
1) 개발자가 자바 객체(엔티티)를 정의합니다.
2) 애플리케이션에서 객체를 저장/조회하는 방식으로 코드를 작성합니다.
3) JPA 구현체가 내부에서 SQL 생성/실행을 처리합니다.
4) 조회 결과도 객체로 매핑해서 돌려줍니다.
5) 결과적으로 반복 SQL 작업이 줄고, 변경 대응이 더 쉬워집니다.

## 실무 포인트
- JPA는 “SQL을 아예 모른다”가 아니라 “반복 작업을 줄여준다”에 가깝습니다.
  - 복잡한 쿼리나 성능 이슈에서는 결국 SQL 튜닝 관점이 다시 등장합니다.
- 면접에서 이렇게 말하면 안정적입니다.
  - JPA는 자바 ORM 표준이고, 하이버네이트가 대표 구현체이며, 객체 중심으로 DB를 다루게 해서 반복 SQL/매핑 코드를 줄여줍니다.
- JDBC 직접 방식의 핵심 단점 2개만 기억하면 됩니다.
  - 반복 작업이 너무 많다
  - SQL 의존적이라 변경에 취약하다

## 용어 정리
| 용어 | 뜻 |
| --- | --- |
| ORM | 객체와 테이블 관계를 매핑해주는 기술 |
| JPA | 자바 ORM에 대한 표준 명세 |
| Hibernate | JPA 구현체(사실상 표준) |
| JDBC | 자바에서 DB 연결/쿼리 실행을 위한 API |
| Entity | 테이블과 매핑되는 자바 객체(개념) |

# Server Basic

## 개요
서버는 클라이언트(브라우저, 앱)가 보낸 요청을 받아서 처리하고, 결과를 응답으로 돌려주는 프로그램(또는 컴퓨터)입니다. 서버를 이해하려면 네트워크, IP와 포트, HTTP 요청과 응답 흐름을 같이 잡는 게 좋습니다.

## 정의
- 네트워크: 여러 컴퓨터/장비가 연결되어 정보를 주고받을 수 있게 해주는 환경
- 클라이언트(Client): 서버에 요청을 보내는 쪽(브라우저, 앱 등)
- 서버(Server): 요청을 받아 처리하고 응답을 주는 쪽
- IP 주소: 네트워크에서 컴퓨터(장비)를 식별하는 주소
- 포트(Port): 한 컴퓨터 안에서 어떤 프로그램(서비스)로 보낼지 구분하는 번호
- 웹 서버(Web Server): HTTP 요청을 받아 응답하는 서버(정적 콘텐츠 전달에 강함)
- WAS(Web Application Server): HTTP 기반으로 동적 로직(로그인, 조회, 정렬 등)을 실행하는 서버(애플리케이션 실행에 강함)

## 핵심 메커니즘(핵심 개념)
1) 요청은 IP와 포트로 도착한다  
IP는 "어느 컴퓨터로 갈지", 포트는 "그 컴퓨터 안에서 어느 프로그램으로 갈지"를 의미합니다.

2) 웹은 보통 HTTP로 통신한다  
브라우저가 HTTP Request를 보내면, 서버가 HTTP Response로 돌려줍니다.

3) API는 약속이다  
서버와 클라이언트가 "이 URL로 이렇게 요청하면, 이렇게 응답한다"를 약속으로 정해 둔 규칙입니다.

## 동작 방식
- 사용자는 브라우저에서 URL을 요청합니다.
- 요청은 인터넷을 타고 서버의 IP로 도착합니다.
- 서버는 포트로 들어온 요청을 받아 처리합니다.
- 서버는 처리 결과를 HTTP 응답으로 돌려줍니다.
- 브라우저는 응답을 받아 화면을 그립니다.

## 전체 흐름
1) 클라이언트(브라우저/앱)가 요청을 만든다  
2) 네트워크를 통해 서버의 IP로 요청이 전달된다  
3) 서버에서 포트로 요청을 받아서 처리할 프로그램이 결정된다  
4) 서버가 요청을 처리하고 응답을 만든다  
5) 응답이 클라이언트로 돌아간다  
6) 브라우저는 응답 데이터를 화면에 렌더링한다

## 실무에서 고려할 포인트
- 서버 개발을 시작할 때는 "요청이 서버까지 어떻게 오고, 응답이 어떻게 돌아가는지" 흐름을 먼저 익히는 게 제일 중요합니다.
- 웹 서버와 WAS는 역할이 겹치기도 하지만, 보통은 정적 처리와 동적 처리를 분리해서 구성하는 경우가 많습니다.

## 용어 정리
| 용어 | 뜻 |
| --- | --- |
| Client | 요청을 보내는 쪽(브라우저, 앱) |
| Server | 요청을 받아 처리하고 응답하는 쪽 |
| Network | 장비들이 연결되어 데이터를 주고받는 환경 |
| IP | 네트워크에서 컴퓨터(장비)를 식별하는 주소 |
| Port | 컴퓨터 안에서 프로그램을 구분하는 번호 |
| HTTP | 웹에서 요청/응답을 주고받는 규칙(프로토콜) |
| Web Server | 정적 콘텐츠 응답에 강한 서버 |
| WAS | 동적 로직 실행에 강한 서버 |
| Tomcat | 대표적인 WAS(서블릿 컨테이너), Spring에서 자주 사용 |
| API | 시스템끼리 통신하기 위한 약속(규칙) |
| REST | API를 설계할 때 지키는 스타일(리소스, HTTP 메서드 활용 등) |
| Postman | API 요청/응답을 쉽게 테스트하는 도구 |
